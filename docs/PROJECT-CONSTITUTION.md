PROJECT-CONSTITUTION

行者学社 PROJECT-CONSTITUTION
1. 项目名称

行者学社

2. 产品定位

活动驱动的人生经历沉淀平台。

完整定位：

活动报名支付系统 + 精美证书传播系统 + 克制型用户运营系统。

3. 核心目标

通过线下活动、证书、足迹、地图和同行关系，帮助用户记录行动经历、获得精神满足、形成长期沉淀，并通过分享吸引新用户。

4. 核心主线
主线一：活动交易闭环

活动创建
→ 发布
→ 报名
→ 支付
→ 二维码
→ 核销
→ 退款
→ 发票
→ 财务

主线二：证书分享闭环

核销
→ 自动生成证书
→ 证书展示
→ 证书分享
→ 朋友圈传播
→ 新用户进入
→ 活动报名

主线三：克制用户运营闭环

用户画像
→ 活动参与
→ 城市足迹
→ 证书记录
→ 邀请统计
→ 订单金额
→ 退款情况
→ 标签备注
→ 温和运营

5. 系统优先级

当前阶段优先级（V2.7.2 线上测试稳定版）：

P1 活动报名支付闭环 ✅ 已完成
P2 订单 / 退款 / 财务 / 发票 ✅ 已完成
P3 CRM 用户运营 ✅ 已完成
P4 真实用户与微信登录 ✅ 已完成
P5 活动产品模型增强 ✅ 已完成
P6 证书模板基础能力 ✅ 已完成（V2.7.3 需完整回归）
P7 行者地图 / 足迹 / 活动地点 ✅ V2.6 已完成基础版
P8 线上部署上线 ✅ V2.7 已完成（腾讯云 MySQL + Nginx + HTTPS）
P9 真实支付（当前不接）
P10 邀请增长（后续版本）


6. 产品原则
活动系统优先
业务闭环优先
可运行优先
克制运营优先
证书分享是传播动力
行者地图是长期身份沉淀
同行者是轻关系连接，不做重社交
活动级邀请优先于全局上下级
后台服务运营，不替代小程序体验

订单：
支付状态、金额、退款、发票

报名：
报名状态、签到、二维码

证书：
参与结果

CRM：
用户关系

7. 禁止事项
禁止：
论坛
社区
直播
短视频
积分商城
复杂等级体系
粗暴分销
过度营销
强社交
低级被动放松
AI 分析系统抢占主线
微服务
DDD
Event Sourcing
MQ
无必要架构升级
无必要性能优化专项
业务状态机重写
破坏报名支付主链路
8. 架构原则
单体优先
本地可运行优先
文档优先
状态机清晰
后端是唯一业务中枢
Admin 不直接操作数据库
WeApp 不直接操作数据库
核心状态流转必须经过后端 service
9. 数据事实源
Registration = 报名事实源
Order = 支付记录
Refund = 退款事实源
Invoice = 发票事实源
QR = 核销事实源
CRM = 运营视图，不改核心状态
10. Agent 治理原则

用户只与 CEO Agent 沟通。

CEO Agent：ChatGPT

负责：

产品规划
需求冻结
系统设计
任务拆解
验收标准
QA 判断

Dev Agent：Claude

负责：

代码实现
接口实现
前端实现
自查（不允许以"我已验证通过"替代 QA Agent 报告）
完成后输出可交给 QA Agent 的检查清单

QA Agent：tools/qa-agent + 豆包/火山方舟

负责：

静态规则
API 回归
LLM 审查
报告输出
独立质量检查（Claude 不允许替代 QA Agent）

正确开发流程：
1. CEO Agent 输出需求文档
2. Claude 执行开发
3. Claude 自查
4. QA Agent 独立检查
5. CEO Agent 根据 QA 报告判断是否修复
6. Claude 修复后重跑 QA
7. QA blocking = 0 后人工验收
8. Git 封版
