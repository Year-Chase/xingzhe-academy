PROJECT-CONSTITUTION

行者学社 PROJECT-CONSTITUTION
1. 项目名称

行者学社

2. 产品定位

活动驱动的人生经历沉淀平台。

完整定位：

活动报名支付系统 + 精美证书传播系统 + 克制型用户运营系统。

3. 核心目标

通过线下活动、证书、足迹、地图和同行关系，帮助用户记录行动经历、获得精神满足、形成长期沉淀，并通过分享吸引新用户。

4. 核心主线
主线一：活动交易闭环

活动创建
→ 发布
→ 报名
→ 支付
→ 二维码
→ 核销
→ 退款
→ 发票
→ 财务

主线二：证书分享闭环

核销
→ 自动生成证书
→ 证书展示
→ 证书分享
→ 朋友圈传播
→ 新用户进入
→ 活动报名

主线三：克制用户运营闭环

用户画像
→ 活动参与
→ 城市足迹
→ 证书记录
→ 邀请统计
→ 订单金额
→ 退款情况
→ 标签备注
→ 温和运营

5. 系统优先级

当前阶段优先级：

P1 活动报名支付闭环
P2 订单 / 退款 / 财务 / 发票
P3 CRM 用户运营
P4 真实用户与微信登录
P5 活动产品模型增强
P6 真实支付与订单增强
P7 证书传播
P8 行者地图
P9 邀请增长
P10 云部署上线

6. 产品原则
活动系统优先
业务闭环优先
可运行优先
克制运营优先
证书分享是传播动力
行者地图是长期身份沉淀
同行者是轻关系连接，不做重社交
活动级邀请优先于全局上下级
后台服务运营，不替代小程序体验
7. 禁止事项

禁止：

论坛
社区
直播
短视频
积分商城
复杂等级体系
粗暴分销
过度营销
强社交
低级被动放松
AI 分析系统抢占主线
微服务
DDD
Event Sourcing
MQ
无必要架构升级
无必要性能优化专项
业务状态机重写
破坏报名支付主链路
8. 架构原则
单体优先
本地可运行优先
文档优先
状态机清晰
后端是唯一业务中枢
Admin 不直接操作数据库
WeApp 不直接操作数据库
核心状态流转必须经过后端 service
9. 数据事实源
Registration = 报名事实源
Order = 支付记录
Refund = 退款事实源
Invoice = 发票事实源
QR = 核销事实源
CRM = 运营视图，不改核心状态
